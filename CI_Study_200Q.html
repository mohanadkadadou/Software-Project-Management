Define Continuous Integration in your own words. What makes it 'continuous' and what is verified in each integration?
عرّف الـ Continuous Integration بكلماتك. ما الذي يجعله 'مستمراً' وماذا يتم التحقق منه في كل دمج؟
Definition: CI is a software development practice where team members integrate their work frequently — at least daily — leading to multiple integrations per day. Each integration is verified by an automated build (including tests) to detect integration errors as quickly as possible.

'Continuous' explained: 'Continuous' in CI means CONTINUAL, not non-stop. A CI server continuously polls the VCS for changes. When changes are detected, it executes a build script. The build does not run constantly — it runs whenever triggered by a commit.

What is verified: Compilation, automated tests, code inspection (static analysis), and potentially deployment — all in the automated build.
CI ممارسة يدمج فيها الفريق عملهم يومياً على الأقل مع تحقق آلي في كل دمج. 'Continuous' تعني متواصل لا لا-يتوقف. الخادم يراقب VCS ويُطلق البناء عند اكتشاف تغيير. يتم التحقق من: compile + اختبارات + فحص كود + نشر.
Q2Definition
What is the difference between a build, a private build, an integration build, and a release build?
ما الفرق بين build وprivate build وintegration build وrelease build؟
Build: A set of activities to generate, test, inspect, and deploy software. Not just compilation.

Private Build: Run locally on developer's workstation before committing. Emulates the integration build. Reduces chance of breaking the integration build.

Integration Build: Run on a separate dedicated machine (not a developer's workstation). Combines all components. Triggered by VCS commits via CI server (CruiseControl).

Release Build: Readies software for release to users. Occurs at end of iteration/milestone. Must include acceptance tests and may include performance/load tests.
Build: توليد+اختبار+فحص+نشر. Private: محلي قبل commit. Integration: جهاز منفصل عند كل commit. Release: نهاية iteration يشمل acceptance tests.

1. Commit Frequently: Small changes → less risk of conflicts, faster detection of issues.

2. Don't Commit Broken Code: Run private build first → protects the team from a broken shared environment.

3. Fix Broken Builds Immediately: Top priority → everyone is blocked by a broken build. Fast fix culture.

4. Write Automated Tests: Computers can't judge code quality → xUnit tests (JUnit/NUnit) provide objective verification.

5. All Tests Must Pass (100%): Any failure = unknown broken state. Partial passing is not confidence.

6. Run Private Builds: Catch integration issues on your own machine before affecting the team.

7. Avoid Getting Broken Code: Don't pull broken code → you'll waste time working around broken issues.

Explain the system reliability math with examples. Why is reliability a 'product' not an 'average'? What does this mean for CI?

The Math:
3 components × 90% = 0.90 × 0.90 × 0.90 = 73% (NOT 90%)
4 components × 95% = 0.95⁴ ≈ 81.5% (NOT 95%)

Why PRODUCT, not average: For the system to function, ALL components must work simultaneously. If component A fails, the system fails — regardless of B and C being perfect. Therefore reliability compounds multiplicatively downward.

Quote: 'True reliability isn't an average; it's a product. It must be built from the lowest possible building block.'

CI Implication: This is precisely why CI requires 100% test pass. Each failing test = one component with <100% reliability. Even one failure degrades overall system reliability. '99%' pass rate across 100 tests means you have unknown failures that compound.
3×90% = 73% وليس 90%. لأن النظام يفشل إذا فشل أي مكون واحد → الموثوقية تتضاعف هبوطاً (ضرب لا جمع). في CI: لهذا 100% اختبارات ناجحة — حتى اختبار واحد فاشل يخفض الموثوقية الكلية.

The Trade-off: Every automated test is an investment that prevents you from running other potentially valuable manual tests. The COST of automation = the bugs you'll miss from not running those manual tests instead.

Key Question: 'If I automate this test, what manual tests will I LOSE?' This forces you to evaluate the value of automation vs. the value of the manual tests you'd replace.

1. Configuration Testing: Testing on a new OS or device = logically equivalent to testing changed support code. High repetition + consistency required → automation wins.

2. Faster Debugging: A daily automated suite finds bugs the day after they're introduced — dramatically reduces debugging time and defect lifetime.

3. Task-Driven Tests: Tests that simulate real user workflows (not single features) exercise more support code and catch more bugs than manual single-feature testing.
كل اختبار آلي مقايضة — تكلفته = الأخطاء التي ستفوتها من عدم تشغيل اختبارات يدوية أخرى. السؤال: 'ما الاختبارات التي سأخسرها؟' 3 عوامل لزيادة القيمة: 1.اختبار على OS جديد 2.اكتشاف أسرع (يوم التالي) 3.workflow كامل للمستخدم.

Q48Pipeline
Explain 'integrating continuously' as a transformation principle. What specific developer behaviors does it require?
اشرح 'Integrate Continuously' كمبدأ تحول. ما السلوكيات المحددة للمطور التي تتطلبها؟
The Principle: Every developer commits working code to the mainline (main branch) at least daily. The CI server is triggered by each commit and runs the full quality pipeline. Small, frequent, complete integrations replace infrequent, large, risky ones.

Why daily minimum: The CI definition states 'each person integrates at least daily — leading to multiple integrations per day.' More frequent = smaller changes = easier to identify what broke = faster fix.

Specific developer behaviors required:

1. Break work into small tasks: Instead of 'implement the full payment module' (2 weeks), break into 'add payment model class' (2 hours), 'add validation logic' (3 hours), etc.

2. Run private build before committing: Get latest from VCS → merge → compile → run tests → then commit. Never commit blind.

3. Commit to mainline, not branches: Long-lived feature branches defeat 'continuous' — changes must be integrated with everyone's work daily.

4. Fix broken builds immediately: If your commit broke the build — fix it RIGHT NOW. Don't commit more code on a broken build.

5. Never commit at end of day without running tests: 'Commit code that you know builds — not untested code at 5pm Friday.'

Cultural shift required: From 'I'll commit when it's perfect' to 'I'll commit when it compiles and tests pass, even if incomplete.'
كل مطور يعمل commit للـ mainline يومياً على الأقل. السلوكيات: 1.مهام صغيرة 2.private build قبل كل commit 3.commit للـ mainline لا branches طويلة 4.إصلاح فوري للبناء المكسور 5.لا commit بدون اختبارات. التحول الثقافي: من 'مثالي' إلى 'يعمل ويختبر.'
Q49Definition
CI relies on version control (VCS) as the single source of truth. List everything that should be stored in VCS and explain why each category matters.
CI يعتمد على VCS كمصدر حقيقة واحد. اذكر كل ما يجب تخزينه في VCS واشرح لماذا كل فئة مهمة.
Everything needed to build and deploy should be in VCS:

1. Source Code: Obviously. The core of what is being built. Without this, nothing works.

2. Database Scripts: Creation scripts, migration scripts, seed data scripts. Allows any environment to recreate the DB from scratch. Solves 'DB Syncing' scenario from Risk 1.

3. Stored Procedures & Triggers: Database logic that must be deployed alongside application code. If out of sync = runtime errors.

4. Build Scripts (Ant/NAnt/Rake): The automation recipes. Must be in VCS so every environment uses the same build process. Without this, builds are person-dependent.

5. Configuration Files: Server settings, connection strings, environment properties. Enables reproducible environments. Without this: 'works in dev, fails in staging.'

6. Test Data: Test fixtures, seed data for test environments. Without this: component tests can't run in a clean CI environment.

7. Infrastructure Scripts (CI config): CruiseControl config files, server setup scripts. Team can rebuild CI environment if server crashes.

What should NOT be in VCS: Compiled artifacts, generated files (these are produced BY the build), sensitive production credentials (use secrets management instead).
في VCS يجب أن يكون: 1.كود المصدر 2.سكريبتات DB (creation+migration) 3.stored procedures+triggers 4.build scripts (Ant) 5.ملفات config 6.test data 7.إعدادات CI. لا يخزَّن في VCS: artifacts المُترجَمة أو ملفات مُولَّدة.
