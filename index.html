Chapter 1
Introducing Continuous Integration
المصطلحات الأساسية · الممارسات السبع · قيمة CI
English Definition

Continuous Integration (CI) is a software development practice where members of a team integrate their work frequently — usually each person integrates at least daily — leading to multiple integrations per day. Each integration is verified by an automated build (including test) to detect integration errors as quickly as possible.

التعريف بالعربي

الـ CI ممارسة تطوير حيث يدمج أعضاء الفريق عملهم بشكل متكرر — عادةً يومياً على الأقل — مما يؤدي لعمليات دمج متعددة يومياً. كل دمج يتم التحقق منه بواسطة بناء آلي يشمل الاختبارات لاكتشاف الأخطاء بأسرع وقت ممكن.

Key Terms — المصطلحات

build
A set of activities performed to generate, test, inspect, and deploy software. Not just compilation.
مجموعة أنشطة لتوليد واختبار وفحص ونشر البرنامج. وليس فقط الـ compile.
continuous
In CI, means continual — the CI server constantly polls the VCS for changes and runs the build script when a change is detected.
في CI تعني متواصل — خادم CI يراقب VCS باستمرار وينفّذ السكريبت عند اكتشاف تغيير. ليس "لا يتوقف".
private build
Running a build locally before committing to the VCS, to reduce the chance of breaking the integration build.
تشغيل البناء محلياً قبل الـ commit لتقليل احتمال كسر بناء التكامل.
integration build
Combining components on a separate dedicated machine — not a developer's workstation.
دمج المكونات على جهاز منفصل ومخصص — وليس جهاز المطور.
release build
Readies software for release to users. Must include acceptance tests. May include performance tests.
يُجهّز البرنامج للإصدار. يشمل acceptance tests وأحياناً performance tests.
quality
A measurable specification, not perception. Metrics: maintainability, extensibility, security, performance, readability.
مواصفة قابلة للقياس وليست إدراكاً. المقاييس: الصيانة، التوسع، الأمان، الأداء، القراءة.
risk
The potential for a problem to occur. A risk that has been realized is known as a problem.
احتمال حدوث مشكلة. الخطر الذي تحقق يُسمى مشكلة.
inspection
Analysis of source code for internal quality attributes. Includes static and runtime analysis (automated aspects).
تحليل الكود المصدري للسمات الجودة الداخلية. يشمل التحليل الثابت والديناميكي.

The 7 CI Practices — الممارسات السبع

1
Commit Code Frequently — الـ Commit بشكل متكرر
Make small changes. Commit after each task. Don't accumulate large changes — break tasks into smaller work items. The main reason developers don't commit frequently is project culture (wanting "perfect" code).
اعمل تغييرات صغيرة. commit بعد كل مهمة. السبب الرئيسي لعدم الـ commit المتكرر هو ثقافة المشروع (الرغبة في كود "مثالي").
2
Don't Commit Broken Code — لا تعمل Commit لكود مكسور
Always run a private build that closely resembles the integration build before committing. This is a team culture expectation.
دائماً شغّل private build يشبه بناء التكامل قبل الـ commit. توقع ثقافي للفريق بأسره.
3
Fix Broken Builds Immediately — أصلح البناء المكسور فوراً
A broken build is anything preventing success: compile error, failed test, DB problem, failed deployment. Fixing it is the top project priority.
البناء المكسور هو أي شيء يمنع النجاح. إصلاحه هو أعلى أولوية في المشروع.
4
Write Automated Developer Tests — اكتب اختبارات آلية
Tests must be in an automated framework like JUnit or NUnit (xUnit). Tests cannot be manual — the CI system must be able to run them without human intervention.
الاختبارات يجب أن تكون بـ framework آلي مثل JUnit أو NUnit. لا يمكن أن تكون يدوية.
5
All Tests and Inspections Must Pass — 100% نجاح الاختبارات
100% of tests must pass. Code with test failures = broken code. An unscrupulous developer might comment out failing tests — coverage tools help detect this.
100% يجب أن ينجح. الكود الذي يفشل في الاختبارات = كود مكسور. مطور قد يحذف اختباراً فاشلاً — أدوات coverage تساعد في الكشف.
6
Run Private Builds — شغّل بناءات خاصة محلياً
Before committing: get latest from VCS, merge your changes, compile, run tests. This ensures what you commit has been verified against everyone else's latest code.
قبل الـ commit: اجلب أحدث كود من VCS، ادمج تغييراتك، compile، شغّل الاختبارات. يضمن أن ما تُرسله تم التحقق منه.
7
Avoid Getting Broken Code — لا تسحب كود مكسور
When the build is broken, don't check out the latest code. Use passive feedback mechanisms (lights, sounds) to know build state. Everyone should know the state of the code in VCS.
عندما البناء مكسور، لا تسحب آخر كود. استخدم آليات ملاحظة passive (أضواء، أصوات). الجميع يجب أن يعرف حالة الكود.

5 Values of CI — قيمة CI

1 Reduce Risks

Defects detected when introduced. Health is measurable. Reduce assumptions by using consistent process.
اكتشاف العيوب فور إدخالها. صحة البرنامج قابلة للقياس.

2 Reduce Repetition

Same ordered process every time. Frees people for higher-value work.
نفس العملية كل مرة. يحرر الوقت للعمل الإبداعي.

3 Deployable Software

Always have a releasable version. No "big bang" integration at end of project.
دائماً يوجد إصدار جاهز للتسليم في أي وقت.

4 Better Visibility

Just-in-time info on build status and quality metrics. Notice trends over time.
معلومات آنية عن حالة البناء ومقاييس الجودة.

5 Greater Confidence

Every build verifies behavior and standards. Developers confident making changes.
كل بناء يتحقق من السلوك والمعايير. ثقة أعلى بالتغييرات.

⚠️ Critical Distinction: Continuous Compilation ≠ Continuous Integration. Setting up CruiseControl to only compile is better than nothing, but it does NOT provide full CI benefits. Real CI adds: automated tests, code inspection, automated deployment, DB integration, feedback notifications.

الفرق الحاسم: الـ compile فقط ≠ CI كامل. CI الحقيقي يضيف: اختبارات، فحص كود، نشر آلي، تكامل DB، إشعارات.
Chapter 2
Reducing Risks Using CI
4 مخاطر رئيسية · سيناريوهات حقيقية · حلول · أدوات
Each risk follows a fixed template: Introduction → Scenario (with named characters) → Solution → Tools
كل مخاطرة تتبع قالباً ثابتاً: مقدمة → سيناريو (بأسماء حقيقية) → حل → أدوات
1
Lack of Deployable Software — غياب برنامج قابل للنشر
Scenario A — "It Works on My Machine" · آدم وجون
English
Adam committed source files but forgot to add new files to Subversion. Build fails on the integration server but works on his machine. Root cause: tight coupling between IDE and build process.
عربي
آدم عمل commit لكن نسي إضافة الملفات الجديدة لـ Subversion. البناء يفشل على الخادم لكن يعمل على جهازه. السبب الجذري: ارتباط وثيق بين IDE والبناء.
Scenario B — Synching with the Database
English
DB team doesn't commit scripts to VCS. Developers fear making schema changes. Can't populate test data. Dev and test environments get out of sync.
عربي
فريق DB لا يضع السكريبتات في VCS. المطورون يخافون تغيير الـ schema. لا يمكن تعبئة بيانات الاختبار. البيئات تتشعب.
Scenario C — The Missing Click · راشيل وجون
English
Manual deployment takes 10–15 min. John forgot to select "precompile JSPs" — runtime errors appear. Deployment is a bottleneck depending on one person.
عربي
النشر اليدوي 10-15 دقيقة يومياً. جون نسي تفعيل خيار — أخطاء runtime. النشر أصبح عنق زجاجة.
Solution / الحل
Use a separate integration machine (CruiseControl). Everything to build goes in VCS — source, DB scripts, triggers, stored procedures. Use Ant/NAnt/Rake scripts. Automate deployment.
جهاز تكامل منفصل. كل ما يلزم للبناء في VCS — كود، سكريبتات DB، triggers، stored procedures. استخدم Ant/NAnt/Rake. أتمتة النشر.
CruiseControlAntNAntRakeSubversion
2
Late Discovery of Defects — اكتشاف متأخر للعيوب
Scenario A — Regression Testing · سالي وكايل
English
Sally (Lead) finds a bug that was fixed 2 months ago has returned. Kyle only ran tests for his changes. Old bug came back (regression) undetected.
عربي
سالي تجد خطأً قديماً عاد. كايل اختبر تغييراته فقط. الخطأ القديم عاد (regression) دون اكتشاف.
Scenario B — Test Coverage · إيفلين ونوح
English
Manager asks if Noah ran tests. He says yes. But she never asked: Did you write tests for new code? What % of code is covered? No way to verify without coverage tools.
عربي
المدير تسأل نوح إن شغّل الاختبارات. يقول نعم. لكنها لا تسأل: هل كتبت اختبارات جديدة؟ ما نسبة التغطية؟
Solution / الحل
Write unit + component tests in JUnit at all layers. Configure Ant to run ALL tests every build. Add code coverage tool. Run everything continuously on every VCS commit.
اكتب unit + component tests بـ JUnit لجميع الطبقات. هيّئ Ant لتشغيل كل الاختبارات في كل بناء. أضف code coverage. شغّل باستمرار.
JUnitNUnitAntCode Coverage Tools
3
Lack of Project Visibility — غياب رؤية المشروع
Scenario A — "Did You Get the Memo?" · إيفلين ونوح
English
Noah (Tester) waiting for the build to be deployed to QA. Evelyn tells him it was deployed 2 days ago. He had no idea — was out of office. Manual communication fails.
عربي
نوح ينتظر نشر البناء للـ QA. إيفلين تخبره أنه نُشر منذ يومين. لم يعلم — كان خارج المكتب. التواصل اليدوي يفشل.
Scenario B — Inability to Visualize · مايلي وآلي
English
Maile (new dev) asks to see UML diagrams. Allie says "we don't do UML, just read the code." No auto-generated diagrams = new developers can't grasp the architecture quickly.
عربي
مايلي (مطورة جديدة) تطلب مخططات UML. آلي ترفض. بدون مخططات آلية، المطورون الجدد لا يفهمون المعمارية بسرعة.
Solution / الحل
Configure CruiseControl to send automatic email + SMS when build fails. Install automated agents to check server availability. Use Doxygen to auto-generate UML diagrams from source code — always up to date.
هيّئ CruiseControl لإرسال إيميل + SMS تلقائياً عند فشل البناء. استخدم Doxygen لتوليد مخططات UML تلقائياً من الكود — دائماً محدّثة.
CruiseControl (email/SMS)Doxygen (UML)
4
Low-Quality Software — جودة برمجيات منخفضة
Code Smell: "A symptom that something may be wrong." Potential defects from poor design, non-standard code, or duplication. / عرض يشير لوجود مشكلة محتملة.
Sub-risk A — Coding Standards · برايان وليندساي
Problem/الحل
Lindsay writes in personal style; 30-page doc ignored. → Replace with 1-page annotated class + Checkstyle/PMD in every build. Fail build on violations.
عربي
ليندساي تكتب بأسلوبها؛ وثيقة 30 صفحة مهملة. → صفحة واحدة + Checkstyle/PMD في كل بناء. فشل البناء عند المخالفة.
Sub-risk B — Architectural Adherence · جين ومارك وتشارلي
Problem/الحل
Developer calls data layer from controller — violates "data layer must never talk to business layer." → JDepend/NDepend in every integration build to detect violations automatically.
عربي
مطور يستدعي data layer من controller — مخالفة معمارية. → JDepend/NDepend في كل بناء لاكتشاف المخالفات.
Sub-risk C — Duplicate Code · ماري وآدم
Problem/الحل
Copy-paste habit. One study: 45% duplication in an entire company. → Use Simian or PMD CPD. Refactor to single shared method. Run continuously.
عربي
عادة copy-paste. دراسة: 45% تكرار في شركة كاملة. → Simian أو PMD CPD. إعادة هيكلة. تشغيل مستمر.
CheckstylePMDJDependNDependSimianPMD CPD
Chapter 3
The Deployment Pipeline
مراحل النشر · الموازنة · موثوقية النظام · المبادئ الخمسة
Definition

A deployment pipeline is an automated manifestation of your process for getting software from version control into the hands of your users. It models the process and allows you to see and control the progress of each change — from commit to production.

تعريف

الـ Deployment Pipeline هو تجسيد آلي لعملية إيصال البرنامج من نظام التحكم بالإصدارات إلى يد المستخدمين. يُنمذج العملية ويتيح رؤية ومراقبة كل تغيير — من الـ commit حتى الإنتاج.

Pipeline Stages — مراحل الـ Pipeline

1
Commit Stage
Compile · Unit tests · Static analysis
Fastest feedback
2
Acceptance Test Stage
Automated acceptance tests · Full system
More confidence
3
Manual Testing
UAT · Exploratory testing
Human validation
4
Production
Release to end users
Highest confidence
The Balancing Act: The pipeline balances two competing forces: (1) Increasing Confidence — later stages give more confidence in production readiness. (2) Faster Feedback — earlier stages give faster info on failures. Objective: eliminate unfit release candidates EARLY + get feedback RAPIDLY.

الموازنة: (1) زيادة الثقة — مراحل متأخرة. (2) سرعة الملاحظة — مراحل مبكرة. الهدف: القضاء على مرشحي الإصدار الفاشلين مبكراً + الحصول على ملاحظات سريعة.

System Reliability Math — رياضيات الموثوقية

0.90 × 0.90 × 0.90
= 73%
3 components × 90% reliable each = only 73% overall system reliability
Why this matters

System reliability is a product, not an average. For a system to work, ALL components must work simultaneously. 4 components × 95% = 0.95⁴ ≈ 81.5%. This is why CI requires 100% test pass — each failing test lowers overall reliability. "True reliability isn't an average; it's a product. It must be built from the lowest possible building block."

لماذا هذا مهم

موثوقية النظام حاصل ضرب وليس متوسطاً. لكي يعمل النظام، جميع المكونات يجب أن تعمل في نفس الوقت. 4 مكونات × 95% = 0.95⁴ ≈ 81.5%. لهذا CI يطلب 100% نجاح الاختبارات — كل اختبار فاشل يخفض الموثوقية الكلية.

5 Transformation Principles — المبادئ الخمسة

1
Establish a Single Source of Truth
Commit everything needed to build and test into version control: source code, DB scripts, config files, build scripts, test data.
ضع كل ما يلزم للبناء والاختبار في VCS: كود، سكريبتات DB، ملفات config، بيانات اختبار.
2
Automate the Entire Build
One script builds and packages the entire system from scratch. No manual steps. Anyone on any machine gets the same result.
سكريبت واحد يبني النظام كاملاً من الصفر. لا خطوات يدوية. نفس النتيجة في أي جهاز.
3
Make the Build Self-Testing
Integrate automated unit, component, and regression tests that run with every build. Build fails if any test fails.
ادمج اختبارات unit وcomponent وregression تعمل مع كل بناء. البناء يفشل إذا فشل أي اختبار.
4
Integrate Continuously
Every developer commits to mainline every day, triggering the automated process. No long-lived branches. Small, frequent integrations.
كل مطور يعمل commit للـ mainline يومياً. لا branches طويلة. دمجات صغيرة ومتكررة.
5
Automate Inspection and Deployment
Quality checks and delivery to test environments are seamless, automatic pipeline steps — not manual steps someone must remember.
فحوصات الجودة والتسليم لبيئات الاختبار هي خطوات آلية في الـ pipeline — ليست يدوية.
Chapter 4
Testing Automation
أنواع الاختبارات · قيد المطور · معادلة الأتمتة

Test Types — أنواع الاختبارات

Unit Tests

Test individual functions/methods. Fast. Written by developers. Tests from the developer's own perspective.

تختبر دوال منفردة. سريعة. وجهة نظر المطور فقط.

Component Tests

Test packages/modules together. Also used to test database integration — schema, stored procedures, test data.

تختبر packages/modules معاً. تُستخدم لاختبار تكامل قاعدة البيانات.

System / Acceptance

Test the entire system end-to-end. Verify from the user's perspective. Prove real value is provided.

تختبر النظام كاملاً. وجهة نظر المستخدم. تثبت القيمة الحقيقية.

⚠️ The Developer's Perspective Limitation:
"Unit tests only test a developer's perspective of the solution to a problem. They have only a limited ability to prove that the application does what it is supposed to from a user's perspective."

Two different things: (A) Proving the solution works in a particular way → Unit tests. (B) Asserting the system solves the actual problem for the user → Only Acceptance tests.

قيد وجهة نظر المطور: Unit tests تثبت (أ) أن الحل يعمل بطريقة معينة. لا تثبت (ب) أن التطبيق يحل المشكلة الفعلية للمستخدم. Acceptance tests ضرورية لـ (ب).
The Automation Trade-off

The most important question isn't if we should automate. It is: "If I automate this test, what manual tests will I lose?" Every automated test is a trade-off — an investment that prevents you from running other potentially valuable manual tests. The cost = bugs you miss from not running those manual tests.

معادلة الأتمتة

السؤال الأهم: "إذا أتمتّ هذا الاختبار، ما الاختبارات اليدوية التي سأخسرها؟" كل اختبار آلي هو مقايضة — استثمار يمنعك من اختبارات يدوية أخرى قد تكون ذات قيمة. التكلفة = الأخطاء التي ستفوّتها.

Factors that increase automation value — عوامل تزيد قيمة الأتمتة

Configuration Testing

Testing on new OS/device = logically equivalent to testing with changed support code. High automation value.
الاختبار على OS جديد يعادل اختبار كود الدعم المعدّل.

Faster Debugging

Daily automated suite finds bugs the day after they're introduced — dramatically reduces debugging time.
مجموعة يومية تجد الأخطاء في اليوم التالي لإدخالها.

Task-Driven Tests

Simulate real user workflows (not single features). More variety, exercises more support code.
محاكاة workflow حقيقي للمستخدم بدلاً من دالة واحدة.
